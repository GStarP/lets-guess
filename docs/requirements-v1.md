# lets-guess 详细需求文档（V1）

## 1. 文档信息

- 文档版本：v1.0
- 更新时间：2026-02-10
- 目标读者：产品、设计、前端开发、测试
- 关联文档：`docs/初步设想.md`

## 2. 背景与目标

将线下“看图猜名并写理由”的社交小游戏电子化，降低出题和游玩门槛，并自动生成可传播的结果长图。

核心目标：

1. 出题人只需上传图片并填写正确名称即可快速出题。
2. 猜题人可在移动端高效完成“选名称 + 写理由”。
3. 系统自动生成结果长图并支持一键下载分享。

## 3. 角色定义

- 出题人：创建题目（上传图片、填写名称、导入导出、清空、开始游戏）。
- 猜题人：对每张图片选择名称，并可填写理由。

两种角色可由同一设备同一用户完成，不涉及账号体系。

## 4. 范围定义

### 4.1 In Scope（V1 必做）

- 纯前端应用（React + TypeScript + Vite）。
- 移动端优先，兼容 PC。
- 三页流程：出题页、游戏页、结果页。
- 本地存储：图片使用 IndexedDB，题目元数据使用 localStorage。
- 题目 JSON 的导入/导出（仅题目数据，不含作答记录）。
- 结果页 Canvas 渲染与下载。

## 5. 全局业务规则

1. 每局最少 2 张图，最多 20 张图。
2. 每张图必须填写正确名称，且名称不可重复。
3. 游戏中名称池在每局开始时随机打乱。
4. 猜题时每个名称只能被使用一次（全局唯一占用）。
5. 理由为选填，不影响“是否可完成本局”的判定。
6. 可进入结果页的条件：所有图片均已选择名称。
7. 游戏页点击“返回”后回到出题页，并清空本局作答状态。

## 6. 页面与交互需求

## 6.1 出题页

### 布局

- 顶部栏：上传、开始游戏、更多。
- 更多菜单项：导入、导出、清空。
- 内容区：图片列表，每项包含图片预览和名称输入框。

### 功能

1. 上传图片
   - 支持常见图片格式（`image/jpeg`, `image/png`, `image/webp`）。
   - 上传后立即展示预览项，并创建对应名称输入框。
2. 名称输入校验
   - 去首尾空白后判空。
   - 与其他名称比较时按“规范化字符串”去重（建议忽略首尾空白与大小写差异）。
   - 出现重复时给出行内错误提示。
3. 开始游戏按钮可用条件
   - 图片数量在 `[2, 20]`。
   - 所有名称均已填写且无重复。
4. 开始游戏行为
   - 初始化作答状态。
   - 生成随机名称顺序。
   - 跳转游戏页并定位到第一张图。
5. 导入
   - 选择 JSON 文件并校验 schemaVersion、字段完整性、图片数据合法性。
   - 导入成功后覆盖当前题目草稿。
6. 导出
   - 导出当前题目 JSON（仅题目数据）。
7. 清空
   - 二次确认后清空题目数据与已上传图片。

### 状态与提示

- 空状态：引导上传图片。
- 校验失败：明确指出失败项（数量不足、名称为空、名称重复）。
- 导入失败：提示失败原因（JSON 格式错误、版本不支持、字段缺失、图片数据损坏）。

## 6.2 游戏页

### 布局

- 顶部栏：返回按钮、已选名称展示区（固定两行高度）。
- 主体区自上而下：当前图片、理由输入框（固定两行高度）、可选名称列表。
- 底部/两侧：上一张/下一张按钮。

### 功能

1. 名称选择
   - 点击标签可选中当前图名称。
   - 点击已选名称可取消当前图选择。
   - 当前图已选 A 时点击 B，直接替换为 B。
2. 名称唯一占用
   - 名称被某张图占用后，在其他图中应呈“不可选”状态。
   - 若替换/取消，名称占用关系实时释放。
3. 理由输入
   - 文本可为空，支持多行输入。
4. 翻页
   - 支持左右滑动与按钮翻页。
   - 翻页不丢失当前图已选名称与理由。
5. 查看结果按钮出现条件
   - 所有图片都已完成“名称选择”。
6. 返回行为
   - 点击返回直接回到出题页，并丢弃本局作答数据。

### 可选名称列表区域

- 占满剩余高度。
- 标签横向排布、可换行。
- 超出区域高度后纵向滚动。

## 6.3 结果页

### 布局

- 顶部栏：返回、正确数（`X/XX`）、下载。
- 内容区：结果长图预览（Canvas 渲染输出）。

### 功能

1. 正确率统计
   - 根据每张图“玩家选择名称”与“正确名称”比对得到正确数。
2. 长图渲染
   - 包含每张图的缩略图、玩家选择名称、玩家理由、对错标记。
   - 错误项不展示正确名称，仅展示错误状态。
3. 下载
   - 点击下载后输出 PNG 文件。
4. 返回
   - 返回出题页。

## 7. 结果长图设计规范（清单速读风）

### 设计目标

- 高信息密度，便于快速浏览。
- 强调“玩家怎么选、怎么解释”，弱化纠错教育感。

### 版式建议

- 画布宽度：固定 1080px（导出时），高度按内容自适应。
- 头部：游戏标题 + 正确数 + 生成时间。
- 列表项（每张图一行卡片）：
  - 左侧：图片缩略图。
  - 右侧：
    - 行 1：`选择：<名称>`
    - 行 2：`理由：<文本或“未填写”>`
    - 角标：`正确` / `错误`（低干扰颜色或图标）
- 底部：品牌/应用名（可选）。

### 视觉要求

- 扁平化、简洁，避免复杂拟物。
- 信息层级清晰：图片 > 选择名称 > 理由 > 对错标记。
- 对错标记“可见但不喧宾夺主”。

## 8. 数据模型与存储要求

## 8.1 IndexedDB（图片）

- 数据库名：`lets-guess`
- Object Store：`images`
- 建议字段：
  - `id: string`
  - `blob: Blob`
  - `mimeType: string`
  - `width: number`
  - `height: number`
  - `createdAt: number`

## 8.2 localStorage（题目元数据）

- Key：`lets-guess:puzzle:v1`
- 建议结构：

```json
{
  "schemaVersion": "1.0.0",
  "updatedAt": 1760000000000,
  "items": [
    {
      "id": "img_001",
      "imageId": "img_001",
      "correctName": "名称A"
    }
  ]
}
```

## 8.3 运行期作答状态（内存态）

- `selectedNameByImageId: Record<string, string | null>`
- `reasonByImageId: Record<string, string>`
- `nameOrder: string[]`

> 本局作答状态不要求持久化；用户从游戏页返回出题页时必须清空。

## 8.4 导入/导出 JSON（仅题目数据）

- 必含：`schemaVersion`、`items[]`。
- 每个 item 必含：`id`、`correctName`、`imageData`（建议 Data URL）或可重建图片数据字段。
- 导入时必须进行 schema 校验与错误提示。

## 9. 非功能需求

## 9.1 兼容性

- 移动端优先。
- 支持现代浏览器最近 2 个主版本（iOS Safari / Android Chrome / 桌面 Chrome/Edge）。

## 9.2 性能

- 20 张图规模下，页面交互应保持可用（无明显卡死）。
- 游戏页翻页与标签选择应即时反馈。
- 结果图生成需有加载反馈（避免用户误以为无响应）。

## 9.3 可用性

- 所有关键操作具备可见反馈（成功/失败/禁用原因）。
- 触控目标满足移动端点击可达性。
- 预留两行高度避免内容填入后布局抖动。

## 10. 异常与边界场景

1. 上传非图片文件：拒绝并提示。
2. 达到 20 张上限后继续上传：阻止并提示。
3. 名称重复：开始按钮禁用并提示重复项。
4. 导入 JSON 非法：阻止导入并提示原因。
5. IndexedDB 不可用或写入失败：提示并给出重试建议。
6. 结果图渲染失败：提示失败并允许重试。

## 11. 验收标准（UAT）

## 11.1 出题页

- 给定仅 1 张图时，开始按钮不可点击。
- 给定 2~20 张图且名称完整且不重复时，开始按钮可点击。
- 给定存在重复名称时，开始按钮不可点击且出现重复提示。
- 点击导出后得到可再次导入的 JSON 文件。

## 11.2 游戏页

- 同一名称不能同时分配给两张不同图片。
- 已选名称可取消，可被其他名称替换。
- 理由为空时仍可完成该图片。
- 全部图片完成名称选择后才出现“查看结果”按钮。

## 11.3 结果页

- 顶部正确数与逐项判断一致。
- 每个条目都显示图片、玩家选择、玩家理由、对错标记。
- 错误条目不显示正确名称。
- 点击下载可得到本地 PNG 文件。
